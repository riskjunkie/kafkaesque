# $KAFKA — Design Brief

## The Idea
A memecoin that's honest about what shitcoin trading actually feels like: you ape in on vague premises with no real reasoning, support vanishes, the goal is almost always unattainable, the community copes and gaslights you while your bag dies, and the whole system stays unknowable. It's *Kafkaesque* — a bureaucratic nightmare where logic doesn't apply. $KAFKA is the first memecoin that admits this instead of pretending to have a roadmap.

**Tone rule:** the honesty is the joke, and it's satire about the *trading experience* — not a literal admission that the token itself has no purpose or exit. Dry, deadpan, bureaucratic humor. Never phrase anything as an actual rug disclaimer.

## Aesthetic
Bureaucratic dread, not cartoon-meme soup. Black/white/red palette, filing-stamp textures, typewriter fonts, rejection-letter copy, courtroom/castle silhouettes. Distinct from dog/frog meme visual language.

## The 4-Phase Roadmap (locked)
Sequenced to match the actual emotional arc of getting into a shitcoin:

1. **Before the Law** — pre-launch teaser / waitlist. From the parable: a man waits his whole life at a door meant only for him, and the gatekeeper never lets him in. Maps to hype phase / whitelist / "the door that's about to open."
2. **The Trial** — launch. Josef K. is accused of a crime that's never explained. Maps to: you're in it now, thrown into the system, nobody explains the rules.
3. **The Castle** — ongoing community phase, anchored by the game (see below). K. spends the whole book trying to reach an authority that never responds. Ambient impossible pursuit; leaderboard as the community's shared cope.
4. **The Metamorphosis** — held in reserve for whatever the "everything changes and it's horrifying" moment ends up being (relaunch, LP burn, some irreversible on-chain event). Not yet defined.

## The Castle — Game Concept
Telegram Mini App + embeddable on the website (same build, two entry points — see Technical Approach).

- **Genre:** endless runner, behind-the-character perspective (3rd-person chase cam, à la Temple Run/Subway Surfers) rather than a side-scroller — revised 2026-07-16. 3 lanes, left/right to switch, up to jump, down to duck.
- **Character:** a cockroach (free nod to *The Metamorphosis* — Gregor Samsa — without needing a separate mechanic for it), rendered back-view since the camera sits behind it.
- **The core gag — the castle sits exactly on the horizon.** In a perspective projection, anything at the vanishing point never grows no matter how far you travel — so "unattainable" falls out of the math for free, rather than needing a hand-rolled rubber-band hack. Cleaner and more thematically honest than the original side-scroll version's fixed-background trick.
- **Leaderboard:** tracks furthest distance run, not "reached." Keeps "closest anyone's gotten" meaningful forever without it ever being a lie.
- **Difficulty:** ramps up with distance (speed, obstacle spawn rate) to keep runs finite and scores meaningful.
- **Art plan:** placeholder art is procedural canvas drawing in the current style. Real art to come from ChatGPT-generated stills — back-view character poses (run x2, jump, duck) swapped as a sprite flipbook, plus 3 background parallax layers (horizon/castle, midground corridor, road tile). "Animation" = code-driven frame-swapping and parallax scrolling, not video — AI image tools only produce stills. Prompts given to user 2026-07-16, art not yet generated.

## Technical Approach
- Single self-contained HTML5 file (canvas + JS, no external framework) — same pattern as prior single-file browser game builds.
- Ship once, host it at one URL.
- **Website:** iframe (or direct link) to that URL.
- **Telegram:** wire the same URL as a Telegram Mini App (`web_app` button on a bot) — separate infra decision, deferred until the game itself is built.

**Obstacle/power-up overhaul + difficulty rework (2026-07-16).** User wanted real gameplay variety and a difficulty curve that never plateaus. Rebuilt:
- **Obstacle roster**, each tied to bureaucratic/trading-absurdity flavor, unlocking progressively by distance so the run starts simple: Filing Cabinet / Rejected Stack (jump, base), Rejection Stamp (duck, base), The Auditor's Desk (jump, slow-approach — "the department that takes forever"), Red Tape (duck, spans 2 lanes), Process Server (jump, fast — "you've been served"), The Auditor (jump, shifts lanes mid-approach — a moving figure forcing a reactive dodge), Court Order Barricade (jump, spans all 3 lanes, no dodging possible), Subpoena Banner (duck, spans all 3 lanes). Each obstacle has a `speedMult` (some close distance faster/slower than the base run speed) — this and the multi-lane spans were the specific "dynamics" requested.
- **Power-ups**: Loophole (∞, brief invincibility — obstacles just pass through), Insider Tip (2×, doubles score for a few seconds), Appeal Filed (§, one-time hit shield, stacks if picked up again). Score was split from `distance` internally — `distance` drives difficulty/unlocks and is never multiplied, `score` is what's displayed/saved and gets the 2× buff — so a scoring buff can't accidentally warp the difficulty curve.
- **Difficulty**: both run speed and obstacle spawn-rate now follow an asymptotic curve (`ceil - (ceil-base)*exp(-distance/k)`) instead of the old hard cap — it keeps climbing toward a high ceiling forever without ever fully plateauing ("harder and harder, goes on and on"), while the ceiling itself keeps it humanly survivable indefinitely rather than becoming literally impossible.
- Multi-lane and lane-shifting obstacles required real geometry work: obstacles now carry a `lanes` array (not a single lane) and moving ones interpolate a fractional lane position that settles onto its final lane well before the player's resolution point, so the reactive-dodge window is always fair.

**Verification note for future sessions:** the sandboxed preview browser throttles `requestAnimationFrame` calls too heavily to reach high distances through real-time play (confirmed again this session — even a 12x dt multiplier didn't help, since the bottleneck is *call frequency*, not per-call delta time). Added a `?debug=1` URL flag that exposes `window.__debug` (`step(ms)` to advance the sim without waiting on rAF, `state()`, `spawnLog()`/`powerupLog()` to see full spawn history, `godMode()` for testing survival past the first hit, `forceStart()`). Used it to simulate 500+ seconds of play in one synchronous call and confirmed: every obstacle type unlocks at its correct distance threshold, the difficulty curve climbs smoothly with no NaN/plateau, all 3 power-ups spawn and apply correctly, and collision/invincibility/shield logic all behave as designed. Zero console errors across extended real rendering of the full roster. This debug hook is harmless to leave in (query-param gated, no-op for real players) — reuse it for future gameplay changes instead of fighting the preview environment's throttling again.

**Damage/health system added (2026-07-16).** User didn't want the run to end on the very first hit. Added:
- **3 HP** (shown as `●●●` pips, top-center HUD), each unsafe hazard hit costs 1 HP instead of ending the run immediately; game over only at 0 HP.
- **Mercy invulnerability** (0.9s) after any hit, separate from the Loophole buff, so a single hazard (or two spawned close together) can't chain-drain multiple HP in one moment. Shown as a red pulsing glow around the character (vs. white for an actual Loophole) so the player can tell "just got hit, briefly safe" apart from "invincible."
- **The Refund** (✓) — new healing power-up, restores 1 HP up to the 3-HP cap. Won't spawn at all when already at full HP (no point offering a wasted pickup).
- **Appeal Filed** (shield) still sits in front of HP in the defense order: shield absorbs a hit with zero HP cost; only once the shield is gone does HP start depleting. So the full defense stack is Loophole (total bypass) → Appeal (absorbs one hit free) → HP pool (3 hits) → game over.

Verified via the debug harness: a stationary bot with no autopilot now takes exactly 3 hits to die (previously 1), confirmed the mercy window prevents double-counting, confirmed Refund heals +1 and hard-caps at 3 across repeated pickups, and confirmed a granted shield absorbs exactly one hit before HP starts dropping. Zero console errors.

**Two feel fixes (2026-07-16, same session):**
- **Collision now judges the visually-eased lane, not the instantly-committed one.** `player.lane` snaps the moment a key is pressed, but the sprite (`player.displayLane`) slides toward it over ~0.1-0.3s. If a hazard resolved mid-slide, the hit was judged against a lane the roach hadn't visually reached yet — exactly "damage when it doesn't look like contact happened." Fixed by using `Math.round(player.displayLane)` for collision instead of `player.lane`.
- **Obstacles no longer visually accelerate as they approach.** `projectDepth()`'s `y` and `halfW` were already linear in depth (constant screen velocity), but `scale` used `Math.pow(d, 1.35)` — a convex curve, so an object's *size* ballooned faster and faster near the camera even though its position moved at a constant rate. That compounding growth read as "speeding up." Changed to a plain linear scale (`0.06 + 0.94*d`). Road/environment rendering never reads `.scale` (only `.y`/`.halfW`), so this affects obstacles and power-ups only — each still approaches at its own constant rate (`speedMult` unchanged), just no longer accelerates within its own approach.

**Real item art wired in (2026-07-16, same session).** User generated all 13 items (9 obstacles + 4 power-ups) as a single grid PNG, `Game Imagery/icons.png` (1516×1037, laid out 4+4+5 across three rows, flat black background per the prompt spec). Since an AI-generated grid is never quite evenly spaced, this is NOT sliced by assumed pixel coordinates — `extractIconSheet()` auto-detects each item's real bounding box via connected-component labeling on the alpha-keyed sheet, sorts the blobs into reading order (row-clustered by Y-gap, then sorted by X within each row), and crops each one out individually. One real wrinkle hit and fixed: small disconnected pieces of a single icon (a stamp's hanging chain, dripping ink, scattered papers, a power-up's sparkle particles) don't touch their main shape at the pixel level, so plain connected-components over-split 13 items into 37 blobs. Fixed with a dilate-then-label approach — morphologically dilate the mask first (bridges small internal gaps) to determine grouping, but still accumulate each blob's actual bounding box from the *original undilated* pixels (so the crop hugs real content, not the dilation padding). Dilation radius 8px was the sweet spot for this sheet (18px over-merged: the stamp fused with the auditor below it, and all 4 power-ups fused into one blob) — tunable via `?dilate=N` in the URL if a future icon sheet needs different tuning; found via `?debug=1` → `window.__debug.iconBlobs()`.

Each item has an anchor mode in `ICON_META`: `ground` (bbox bottom sits on the road — cabinet, stack, deskSlow, server, auditor, barricade), `hang` (bbox top hangs above the road — stamp, redtape, subpoena), or `float` (bbox center floats above the road, with a glow — the 4 power-ups). Span items (redtape/barricade/subpoena) size to the obstacle's actual lane-span width rather than a fixed width, matching the old procedural versions. `drawObstacle`/`drawPowerup` try the real sprite first and fall back to the original procedural vector art if a sprite is missing — game never breaks from a missing/failed asset. Visually verified per-item via debug placement helpers (`placeObstacle`/`placePowerup`/`clearField`, godMode to avoid dying mid-inspection) — ground/hang/float anchoring all correct, span items correctly fill their lane width, zero console errors. The Court Order barricade and Appeal scroll in particular look excellent in-game.

**Difficulty rebuilt around elapsed time + real "speed changes" bug fixed (2026-07-16).** User reported (twice, emphatically) that individual obstacles still seemed to speed up/slow down as they approached, despite the earlier linear-scale fix. Root cause found by reading the code, not guessing: `o.depth += depthSpeed * o.speedMult * dt` read the *global* `depthSpeed`, which is recalculated every frame as the difficulty curve climbs — so every obstacle already in flight silently sped up mid-approach as the session ramped, on top of whatever `speedMult` variety it already had. Fixed by locking in `o.speed = currentDepthSpeed(elapsedSec) * def.speedMult` once at spawn time; an obstacle's own closing speed is now provably constant for its entire flight regardless of what the global curve does afterward (verified directly: placed a test obstacle with `speed:0.5`, stepped the sim while the global curve visibly climbed in the background, confirmed the obstacle's own `.speed` field and its depth-per-step never wavered). Same fix applied to power-ups.

While rebuilding this, the whole difficulty system moved from a `distance`-based formula (circular — distance is a downstream effect of speed) to `elapsedSec`-based, so it can be tuned directly against real time targets: **user wants a run easy to survive to ~1 minute, hard to survive to 3-4 minutes.** New asymptotic curves (`currentDepthSpeed`/`currentSpawnGap`, both `ceil - (ceil-base)*exp(-t/timeConst)`): speed 0.42→2.3 over a ~100s time constant, spawn gap 0.9s→0.3s over ~80s. Verified via the debug harness against checkpoints at t=30/60/120/180/240s — depthSpeed climbs from 0.9 to 2.13 across that range, matching hand-calculated values exactly, continuing to climb (never truly plateaus) but bounded by the ceiling. Obstacle unlock thresholds also switched from distance to elapsed time and pulled much earlier (5s/12s/20s/30s/42s/55s instead of the old 150-1350 distance units) so the full 9-obstacle roster is live within about a minute — directly addresses "I want to see them all, not just stamps and desks."

**Plaque removed, second pass at the speed-consistency complaint (2026-07-16, later same session).** User asked to remove the "0 PACES CLOSER" plaque (done, cleanly — no leftover references) and reported the speed-inconsistency issue a third time despite the spawn-lock fix. **Plaque removed; real bug found in duck-obstacle clearance (2026-07-16, later still same session).** User removed the "0 PACES CLOSER" ask (done) and, critically, connected two symptoms that turned out to share a root cause: "hit registers when an item appears to be missed." Two real fixes came out of this:
- **The jump/duck safety check is a pure boolean** (`player.ducking` / `!player.grounded`) — it has never cared where anything is drawn on screen. So if a duck-required obstacle didn't visually read as "that's at head height, duck," a player could easily jump instead (or misjudge), get hit correctly-per-logic, and it'd feel wrong. Fixed at the root per the user's own suggested direction: raised duck-obstacle clearance dramatically. But first found a **real anchor-math bug**: `hangY` was being applied to the TOP of each hanging sprite, not the bottom — meaning the actual clearance gap (the only part that matters for "can I duck under this") drifted per-item based on each sprite's own natural height rather than being a controlled value. Fixed the anchor math (`dy = hangY - bbox.maxY*sf`, bottom-anchored like ground items are) and raised the clearance itself from ~56-58 to ~135-145 world units — well above the roach's ducked head height (~107, from `TARGET_BBOX_W=115` × duck-squash 0.62) and clearly below its standing height (~172), so it's now geometrically unambiguous, not just "hopefully readable." Procedural fallback shapes (`drawStamp`/`drawRedTape`/`drawSubpoena`) updated to match. Visually verified — the subpoena banner in particular now hangs dramatically high with a large, obvious gap above the roach.
- **Collision resolution switched from a depth window to an exact crossing check.** The old `Math.abs(o.depth - PLAYER_DEPTH) < 0.045` could stay "true" across multiple frames at high speed (a single frame's depth step can exceed the window's own width once depthSpeed climbs past ~1.3), so the exact instant it fired — and therefore where the obstacle visually was when judged — could land almost anywhere within a ~26px band rather than lining up with the roach. Replaced with `prevDepth < PLAYER_DEPTH && depth >= PLAYER_DEPTH`, which fires on the exact frame of crossing regardless of step size. Applied the same fix to power-up pickups (with a small ±0.03 tolerance kept there since missed pickups aren't punishing). Re-verified the full obstacle roster and a 2-minute stress run afterward — zero console errors, collision still ends runs correctly.

**Jump replaced with sustained flight (2026-07-16, later still same session).** User wanted "fly indefinitely instead of jumping." Replaced the old fixed-duration ballistic jump (`JUMP_SPEED` one-shot impulse + gravity) with a hold-based flight model: holding ↑ climbs at a constant `FLY_ASCEND_SPEED` (420 units/s) up to `FLY_MAX_HEIGHT` (210, higher than the old jump's ~136 apex) and *holds there indefinitely* for as long as the key is held — not timer-based, genuinely unlimited. Releasing lets gravity take over from whatever height/velocity it was at, giving a natural continued-rise-then-fall if released before the cap. `jump()` was renamed `flyStart()`/`flyEnd()` (mirrors the existing `duckStart()`/`duckEnd()` hold pattern) and wired the same way into keyboard (keydown+keyup on Space/ArrowUp/W) and the D-pad up button.

**Balance fix baked in, not left as a known issue:** the existing air-hazard safety check is a pure boolean (`player.ducking`), independent of `grounded`/height — so under the old brief ~0.85s jump, holding both jump and duck could stack full immunity to both hazard types for less than a second, a minor quirk. Under genuinely *indefinite* flight, that same stacking would have been severe (hold both keys forever, take zero damage ever). Fixed by making `duckStart()` cancel `thrusting` — flight and duck are now mutually exclusive; pressing duck while flying immediately cancels the climb and starts a fall. Verified directly: dispatched a real `ArrowUp` keydown, confirmed height climbs to exactly 210 and holds indefinitely (`vy` drops to 0 at the cap, height unchanged across dozens of further steps) while still flagged `thrusting`; dispatched `keyup`, confirmed smooth gravity descent back to `grounded:true`; dispatched `ArrowDown` mid-flight-at-cap, confirmed `thrusting` flips to `false` and `ducking` to `true` in the same frame, with height immediately starting to fall. Zero console errors.

**Flight-pose art delivered and wired in (2026-07-17).** User generated `cockroach_fly1.png`/`cockroach_fly2.png`. Two things to know for future sessions on this art pipeline:
- **Flat black background, not a vignette** — unlike the original 4 poses. **Barricade/red tape height-only reduction (2026-07-17).** User reported both span obstacles were tall enough up close to blot out whatever was spawning near the horizon behind them. Since they'd been scaled uniformly to match the lane-span width (aspect-ratio-locked), and the source art is tall relative to its width, height came along for the ride. Added `heightScale` to `ICON_META` (barricade 0.55, redtape 0.6) and split `drawIconSprite`'s scale factor into `sf` (width, unchanged) and `hf` (height, `sf * heightScale`) — every Y-axis calculation (ground anchor, hang-clearance anchor) now uses `hf` instead of `sf`, so the anchor edge itself (the road for barricade, the duck-clearance line for redtape) stays exactly where it was and only the object shrinks upward off that fixed edge. This deliberately doesn't disturb the duck-clearance work from the previous round. **⚠️ SPEED ISSUE — DEFINITIVE ROOT CAUSE (2026-07-17, final). This SUPERSEDES every earlier speed entry above, including the "linear scale" fix from earlier the same day, which was aimed at the wrong quantity.**

User reported a 5th time, with the decisive new detail: items appear to **slow down as they get CLOSER**. That detail is what cracked it.

**The actual principle everyone had missed:** the eye judges approach speed *relative to an object's own apparent size*, not in raw pixels — roughly, body-lengths per second. Under the old linear projection an obstacle moved a constant 2.96 px/frame while growing ~8× in size, so near the player it covered ~8× fewer of its own body-lengths per second than it did at the horizon. It therefore **visibly decelerated the entire way in**. Constant pixel velocity is not constant apparent speed — it is guaranteed deceleration whenever the object is also growing.

This is why *four* prior rounds of tuning the scale exponent (`d^1.35` → `d^1` → `d^0.65` → back to linear) could never work: with position linear in depth, perceived speed is constant only if scale is constant — i.e. only with no depth cue at all. The exponent was never the free variable.

**The fix — exponential projection with size strictly proportional to depth-below-horizon.** In `projectDepth`:
```
yOff  = Y_SPAWN * exp(PERSP_K * d)      // Y_SPAWN = 18, PERSP_K = ln(ROAD_SPAN/Y_SPAWN) ≈ 2.80
t     = yOff / ROAD_SPAN
y     = HORIZON.y + yOff ;  halfW = HALFW_BOTTOM * t ;  scale = t
```
Because `d(yOff)/dd = PERSP_K * yOff` and `scale ∝ yOff`, the ratio **(screen velocity)/(apparent size) = PERSP_K * ROAD_SPAN — a constant**. Apparent speed is now mathematically invariant across the whole approach while the object still looms correctly. Width, scale and position all derive from the single `yOff` term, so road, lane lines and every item share one exact perspective (previously halfW, y and scale each followed a *different* curve — ratios 50×, 1×, 16× — which is why nothing ever quite agreed).

**Measured after the change** (60fps frames, one obstacle, full flight): px/frame rises 0.51 → 6.39 (12.5×) while scale rises 0.062 → 0.771 (12.4×), giving perceived speed **8.29, flat, at every sampled depth** (min 8.29 / max 9.12, the max a single frame-timing blip). Before: perceived speed fell ~8× from spawn to player.

**Second real bug found and fixed in the same pass — the road texture was scrolling BACKWARDS.** Scroll was addressed as `(worldScroll + d)`; a fixed patch of road has depth `const + worldScroll`, so its identity is `(d − worldScroll)`. With the `+`, texture features migrated toward *smaller* depth — the road surface streamed *away* from the player while the rungs and obstacles came toward them. Now `(dA − worldScroll) * REPEATS_PER_D`. Rungs and obstacles already used the correct convention, so ground, rungs and items finally all flow together.

**Knock-on changes required by the new projection:**
- `PLAYER_DEPTH` 0.93 → **0.9736**, so the roach still sits at y≈447 (verified 446.9, scale 0.929 vs 0.94 before — layout unchanged). Recompute if `Y_SPAWN`/`ROAD_SPAN` change.
- `drawRoad` rewritten. The road can no longer be sliced in depth space, because reaching the horizon (`yOff→0`) is `d→−∞`. It now slices **uniformly in screen rows** via a new `roadRow(off)` helper (`halfW = HALFW_BOTTOM * off/ROAD_SPAN` — exact linear perspective, edges converge precisely on the vanishing point), 46 bands, with each band's texture coordinate derived back through `d = ln(off/Y_SPAWN)/PERSP_K`. Uniform *depth* slices would have made near bands enormous and chunky.
- Rungs now iterate `i = −8…7` (the road reaches the horizon, well above depth 0) and compute `yOff` directly rather than via `projectDepth`, which clamps `d` at 0.
- Spawn depth `-0.03` → `0` for obstacles and power-ups (a negative depth clamps and would render frozen at the spawn row for a few frames).
- `projectDepth` clamp 1.18 → 1.2; despawn threshold unchanged at 1.2 (an object is only fully off-screen near d≈1.18 under this mapping).

**Re-verified after all of the above:** ground hazard while grounded costs HP (3→2); flying is safe from ground hazards; ducking is safe from air hazards; HP drain fires game over; menus/leaderboard unaffected; zero console errors. Visually the perspective is now geometrically correct — road converges to a true point, distant items are properly small, real depth.

**For any future session: do not touch the scale curve to chase a speed complaint.** The invariant to preserve is `(screen velocity) / (apparent size) = constant`, which holds only while `y − HORIZON.y` is exponential in depth AND `scale`/`halfW` stay strictly proportional to that same `yOff`. Changing any one of those three independently will reintroduce the illusion.

## PERMANENT FIX for the recurring "artwork can't load" / file:// issue (2026-07-19)

User hit the tainted-canvas warning a further time, from opening `index.html` (with the game iframed inside it) directly rather than through `Start Game.command`. Confirmed `Start Game.command` itself has no problem (executable, no Gatekeeper quarantine flag) — this was purely a "which file did I open" workflow trap, and repeating "use the launcher" a fourth time wasn't going to fix that. So the underlying limitation was removed instead of worked around again.

**Root fix: stopped needing `getImageData()` at runtime at all**, for the common case. Three functions — `keyOutBackground()` (character chroma-key), `stripRoadLines()` (road stripe removal), `extractIconSheet()` (icon-sheet blob extraction) — all did real pixel-level processing in-browser every time the page loaded, which is exactly the operation `file://` blocks. New `/Users/allenpelletier/Desktop/Kafka/bake_assets.py` (numpy + Pillow, no scipy available so connected-component labeling is a hand-rolled 4-connected flood fill) replicates all three algorithms **pixel-for-pixel** offline — same corner-sampling, same inner/outer alpha thresholds, same dilate-radius-8 blob merging, same reading-order sort — and writes the *results* as static files into `Game Imagery/baked/`: 9 pre-keyed character PNGs, 1 cleaned `roadTex.png`, 13 individual `icon-<name>.png` files, plus `manifest.json` (character content bounding boxes; icon bboxes aren't stored since they're always `{5, w-5, 5, h-5}` by the fixed padding convention, so they're derived from the loaded image's own `naturalWidth`/`naturalHeight` instead).

`the-castle.html` now tries the baked file first via a plain `<img>` load (never restricted under `file://` — only *reading pixels back out* of a canvas is) using bounding boxes embedded as a `CHAR_BBOX` JS constant (copy-pasted from the manifest, since `fetch()` on a JSON file is *also* blocked under `file://`, same class of restriction as canvas taint). **Only if a specific baked file is missing** does it fall back to loading the raw source image and running the original runtime processing function on it — so the old code path, and the existing `assetsBlocked` warning banner, are fully intact as a safety net (e.g. if new source art is dropped in without re-running the bake script).
